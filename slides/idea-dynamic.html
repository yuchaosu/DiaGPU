SpMV & SpMSpM: the two modes of Hamiltonian simulation

With ℏ = 1, the state advances by |ψ(t+Δt)⟩ = e−iHΔt|ψ(t)⟩. Both paths evaluate the same polynomial in H acting on the state. The only question is how often you reuse the operator:

Single trajectory · no reuse → SpMV

  • Stay matrix-free: apply H to the state vector step by step; H stays banded, no fill-in.
  • Nothing to amortize, so this is the right choice for one (or a few) states.
  • Krylov-subspace methods evaluate the polynomial exactly this way.1
e−iHΔt|ψ⟩ ≈ Σk ck Hk|ψ⟩
y = H · x  (per step)

Operator reused → SpMSpM

  • Build the propagator once as a matrix U (truncated Taylor2); forming Hk is sparse × sparse and fills in.
  • Then each apply is a cheap y = U·x, and it wins once the build amortizes over many states / steps / analyses.3
  • Costlier to build, but the operator is reusable.
e−iHΔt ≈ Σk=0K (−iHΔt)k/k! = U
y = U · x
Both regimes appear in real simulation pipelines, and both are bandwidth-bound on banded H, so we build diagonal-aware kernels for each.

The problem: general libraries pay for generality

  • Dimension is 2q: at q=24 that's a 16.7M × 16.7M operator. Memory bandwidth is everything.
  • General sparse libraries (cuSPARSE CSR) treat H as an arbitrary sparse matrix and pay for it: per-nonzero column indices, gather/indirection, scattered memory.
  • 8 B per nonzero (value + 32-bit index), random access, and at q≥21 they fall back to 64-bit indices.
Operator has 22q elements: a dense store is hopeless (2.3 PB at q=24).
But physics Hamiltonians are not arbitrary: their structure is fixed and known.

Key observation: they're banded

Why are they banded?
  • A local Hamiltonian is a sum of Pauli strings H = Σj cj Pj, each a tensor product of {I,X,Y,Z}.
  • Each string maps to one fixed diagonal: I,Z stay on it, every X/Y on qubit k shifts the column by ±2k, the same offset for every row.
  • A few-body H has only a handful of strings: nonzeros land on a small set of fixed diagonals. So store by diagonal: one offset + one value array each.
  • No per-nonzero column index: 4 B/nnz vs CSR's 8 B/nnz, and access is a regular, coalesced stream, no gather.
This is the whole idea: the Pauli structure of a local Hamiltonian guarantees a fixed, banded set of diagonals, trade generality for that structure and build GPU kernels that stream diagonals.

Offset is stored once per diagonal, so the per-nonzero index disappears: half the footprint, 2× less memory traffic.

Diagonal tensor-core SpMV

  • Each diagonal is a constant offset: row i pairs with x[i+off], a contiguous, coalesced stream, no gather.
  • Lift the band into a dense K×N Matrix (one zero-padded diagonal per row), then per 16-row tile feed A = Recon slice and B = shifted x into a raw mma.sync.m16n8k8.tf32.
  • Register-direct PTX: operands stream global→registers and the 16 outputs are read straight from accumulator registers.
Compare to CSR: scattered gathers become contiguous coalesced loads and the MMA absorbs the multiply-accumulate.

Diagonal-format matrix multiply

  • Stored by diagonal, A and B are each a handful of (offset, value-array) pairs, the rest of the N×N matrix is structurally zero.
  • The full N×N×N dense multiply collapses to a few diagonal × diagonal products: one shifted vector multiply-add per (da,db) pair, no index search.
  • Every output diagonal stays in the band, there is never any work off the diagonals that {da+db} can reach.

Current diagonal SpMSpM algorithm

  • Every (da,db) pair is an independent thread that adds its product into C.
  • Many pairs cross to the same C-diagonal (red crossings → C±1), they hit one cell at once.
  • Collisions force atomicAdd → serialized writes, the bottleneck of the Haque baseline.

SpMSpM Kernel:

  • Dense ↔ diagonal: H[i,j] lives on diagonal d = j−i at position p = min(i,j), each diagonal is one contiguous array.
  • Kernel: one block owns one C-diagonal group; it loads the feeding A/B diagonals into shared memory. Reduction stays on-chip, no atomics.
  • Back to dense: the accumulated C diagonals scatter to C[i,j].

TF32 fidelity:

  • Spin chains & QUBO (tsp, maxcut, tfim, heis) match FP64 to 100.0000% at displayed precision.
  • Only the wider molecular bands drift in the 6th decimal: worst case BeH q12 = 99.9996% of FP64 (≈ 4×10−6).
  • TF32's per-element rounding averages out over the evolution; the aggregate state stays effectively exact.
  • Higher throughput, high accuracy: the TF32 tensor-core path runs far faster than FP64 while staying accurate to ≥ 99.9996%.
Takeaway: TF32 buys the tensor-core throughput at no physical cost(below any measurable observable).

Summary & what's next

Summary

  • Local Hamiltonians are bandedstore by diagonal, then stream diagonals on the GPU instead of paying for generality.
  • Tensor-core SpMVreconstruct the band into dense TF32 tiles → 3.9–5.6× over cuSPARSE (avg 4.7×), fidelity ≈ FP64.
  • Diagonal SpMSpMhost-planned pair schedule + per-block C-groups, no atomics → 5.8–7.2× over cuSPARSE (avg 6.4×), 1.8× over Haque.
