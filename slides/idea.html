SpMV & SpMSpM in Hamiltonian simulation

Hamiltonian simulation, evolving a quantum state forward in time, repeatedly applies the time-evolution operator

|ψ(t)⟩ = e−iHt |ψ(0)⟩ ≈ ( Σk (−iH Δt)k/k! )steps |ψ⟩

Two sparse kernels do all the work in that inner loop:

SpMV: apply H to a state

y = H · x
  • Hot loop of time stepping: each step is one y = Hx.
  • H is 2q×2q sparse; x is the state vector.
  • Sets the cost of the entire trajectory.

SpMSpM: multiply two operators

C = H · H
  • Builds powers Hk and propagator U = Σ ckHk.
  • Computed once, then reused across many steps.
  • Sparse × sparse fills in; cost grows with the band.

The problem: general libraries pay for generality

  • Dimension is 2q: at q=24 that's a 16.7M × 16.7M operator. Memory bandwidth is everything.
  • General sparse libraries (cuSPARSE CSR) treat H as an arbitrary sparse matrix and pay for it: per-nonzero column indices, gather/indirection, scattered memory.
  • 8 B per nonzero (value + 32-bit index), random access, and at q≥21 they fall back to 64-bit indices.
Operator has 22q elements: a dense store is hopeless (2.3 PB at q=24).
But physics Hamiltonians are not arbitrary: their structure is fixed and known.

Key observation: they're banded

Why are they banded?
  • A local Hamiltonian is a sum of Pauli strings H = Σj cj Pj, each a tensor product of {I,X,Y,Z}.
  • Each string maps to one fixed diagonal: I,Z stay on it, every X/Y on qubit k shifts the column by ±2k, the same offset for every row.
  • A few-body H has only a handful of strings: nonzeros land on a small set of fixed diagonals. So store by diagonal: one offset + one value array each.
  • No per-nonzero column index: 4 B/nnz vs CSR's 8 B/nnz, and access is a regular, coalesced stream, no gather.

What is diagonal tensor-core SpMV?

The apply y = Hx with H stored by diagonal, run on the GPU tensor cores (TF32 MMA) instead of the usual CUDA-core scalar loop.

  • Each diagonal of H is a constant offset: row i pairs with state entry x[i+off].
  • Pack the diagonal values into the A operand and the shifted state slices into B, then issue a raw mma.sync.m16n8k8.tf32, the apply y = Hx lands on the tensor cores.
  • Hand-written PTX (not the nvcuda::wmma wrapper) so we control exactly which lanes carry the band, no library overhead.
Why it's fast: the diagonal layout turns scattered CSR gathers into contiguous, coalesced loads, and the MMA absorbs the multiply-accumulate, so the kernel runs at memory speed while the tensor cores hide the arithmetic.

Accuracy: "fast" doesn't cost fidelity

  • SpMV CUDA-core path is exact fp32 (relerr 0 on representable inputs).
  • SpMSpM is bit-exact (max|C − H@H| = 0).
  • Even the TF32 evolution stays effectively exact: infidelity 1e-9 to 6e-9 over 1000 steps to q24.
Framing: over a full 1000-step trajectory at 16.7M dimension, the accumulated infidelity is ~10−9, below any physical observable. We report it as exact.

When does it win? One number.

num_diags / ( 2 · avg_nnz_per_row ) ≲ 2
≲ 2: we win

Heisenberg (1.8), TFIM, Bose–Hubbard, QUBO.
Narrow, dense bands, regular streaming dominates.

≫ 2: cuSPARSE wins

BeH (4.8), O₂ (8.4): wide bands that are 86–93% explicit zeros. Diagonal streams the padding; cuSPARSE skips it.

The win is regular branch-free streaming, not raw byte count, confirmed three ways (compressed-diagonal, ELL, 2D tiling all lost). These kernels are latency-bound.

Summary & what's next

The idea, distilled

  • Diagonal layout + tensor-core SpMV: ~2× cuSPARSE to q24, TF32 fidelity ~1e-9.
  • gather_flat SpMSpM: 1.2–2.2× over HM, 58–446× over cuSPARSE, bit-exact.
  • Operator build: 1.25–2.5× over HM, 13–470× over cuSPARSE.
  • Clean win criterion: num_diags/(2·avg_nnz) ≲ 2.

Next (beyond this basic version)

  • Symmetric / compressed store, already extends SpMSpM reach (this deck deliberately shows the basic kernel).
  • Block-diagonal + batched SpMM, fill the MMA tile with multiple state vectors (block-Krylov) so the tensor cores do useful work per load.
  • Pauli-basis SpMSpM, represent H by its Pauli terms (~280× more compact), build operators algebraically to dodge the Hk fill-in wall.

Bottom line: for local-Hamiltonian dynamics, knowing the matrix is banded is worth ~2× on the hot loop, at TF32 fidelity that's indistinguishable from exact.
